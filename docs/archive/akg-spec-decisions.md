# AKG Spec — Decisions Log

Decisions are logged in the order they were made, following the suggested order of attack from the considerations doc.

---

## Step 1 — Node/Edge Data Model

### 1.1 — Edge identity
**Decision:** Edges do not have an `id` field. The natural key for an edge is the `from_node + relation + to_node` triple. This is sufficient for unique identification and avoids redundant fields.

### 1.2 — Edge mutability
**Decision:** Edges are mutable. `strength`, `confidence`, and `meta` can be updated after creation. Rationale: strength is intended to drift over time as an agent's confidence in a relationship changes — this is core observability into how the knowledge graph evolves, analogous to how real memory strengthens or weakens through reinforcement.

### 1.3 — Versioning
**Decision:** Both nodes and edges carry a `version: uint32` field, incremented on every mutation. Consistent versioning across all entities simplifies the mental model and provides the conflict detection needed by the WAL and merge logic.

### 1.4 — Type system
**Decision:** The binary format is agnostic about the values stored in `type` (nodes) and `relation` (edges). Any string is valid at the format level. The default type taxonomy (`entity`, `decision`, `preference`, etc.) and the `_type_registry` node are SDK-level conventions, not format requirements. The spec will note that custom types are always valid and the registry is informational, not prescriptive.

### 1.5 — Canonical schemas

```
Node {
  type:       string
  title:      string
  body:       string
  meta:       map<string, any>
  tags:       string[]
  created_at: timestamp
  updated_at: timestamp
  version:    uint32
}

Edge {
  from_node:  string
  to_node:    string
  relation:   string
  strength:   float
  confidence: float | null
  meta:       map<string, any>
  created_at: timestamp
  updated_at: timestamp
  version:    uint32
}
```

> Note: `id` is not a field nodes carry about themselves. A node's address in the storage layer is its key in the sorted store (e.g. `n:{type}:{id}`), where the id component is system-generated by the SDK.

## Step 2 — MessagePack Encoding Strategy

### 2.1 — Encoding format
**Decision:** MessagePack, using **maps** (not arrays). Field names are stored alongside values in every serialized record. Rationale: the size overhead is negligible at AKG's target scale (roughly 5-6MB across 100k nodes), and maps are forward-compatible — a reader encountering an unknown field from a future version simply ignores it and carries on. Arrays would break positional decoding the moment any new field was added.

### 2.2 — Timestamps
**Decision:** Unix microseconds as a plain 64-bit integer. Simple, universally understood, no special MessagePack extension type required, and microsecond precision is more than sufficient for AKG's use case.

### 2.3 — Required fields and defaults
**Decision:** Only the fields necessary to make a record meaningful are required. Everything else has a defined default.

- **Node:** `type` and `title` are required. Defaults: `body` → `""`, `tags` → `[]`, `meta` → `{}`, `version` → `1`
- **Edge:** `from_node`, `to_node`, and `relation` are required. Defaults: `strength` → `0.5`, `confidence` → `null`, `meta` → `{}`, `version` → `1`

A reader encountering a missing optional field must apply the default silently, not reject the record.

### 2.4 — Edge weight split: strength and confidence
**Decision:** The single `weight` field from the reference doc is replaced by two fields:

- `strength: float` — required, default `0.5`. How important or central the relationship is.
- `confidence: float | null` — optional, default `null`. How certain the agent is that the relationship is real/correct. `null` means no judgment has been made yet, which is distinct from `0.5` (uncertain but evaluated). This distinction is useful signal for consolidation agents.

Rationale: collapsing strength and confidence into one float loses information. A relationship can be highly important but poorly evidenced, or well-evidenced but loosely connected. Keeping them separate gives consolidation agents richer signal. `confidence` is optional so simple agents can ignore it entirely without penalty.

### 2.5 — Canonical schemas (revised)

```
Node {
  type:       string              // required
  title:      string              // required
  body:       string              // optional, default ""
  meta:       map<string, any>    // optional, default {}
  tags:       string[]            // optional, default []
  created_at: timestamp           // unix microseconds, uint64
  updated_at: timestamp           // unix microseconds, uint64
  version:    uint32              // optional, default 1
}

Edge {
  from_node:  string              // required
  to_node:    string              // required
  relation:   string              // required
  strength:   float               // optional, default 0.5
  confidence: float | null        // optional, default null
  meta:       map<string, any>    // optional, default {}
  created_at: timestamp           // unix microseconds, uint64
  updated_at: timestamp           // unix microseconds, uint64
  version:    uint32              // optional, default 1
}
```

### 2.6 — `created_at` / `updated_at` read vs write semantics
**Decision:** `created_at` and `updated_at` are always written by conformant writers — omitting them is a writer error. They are not required on read — a reader encountering a record without them applies a default of zero rather than rejecting the record. This follows the general principle from 2.3: writers are strict, readers are lenient.

## Step 3 — Key Layout and Index Design

### Core principle
The sorted prefix key approach collapses what would be a complex multi-structure system in a traditional database into one idea: **sort order is architecture**. Indexes are not separate structures — they are the sort order of the data itself. Adding a new index means writing new prefixed keys alongside existing writes. No migrations, no separate index rebuilds.

### 3.1 — Node primary key
**Decision:** `n:{type}:{id}`

All nodes of the same type are physically adjacent on disk. "Give me all `decision` nodes" is a prefix scan on `n:decision:` — no separate type index needed.

ID constraints (spec defines constraints only, not generation strategy):
- Opaque SDK-generated string
- No `:` characters (reserved as delimiter)
- Maximum 64 characters
- Must be unique within the file

### 3.2 — Edge primary key + inverted index
**Decision:** `e:{from_node}:{relation}:{to_node}` for outbound edges, with an inverted entry `ei:{to_node}:{relation}:{from_node}` written atomically on every edge write.

Outbound lookup ("what does this node point to?") and inbound lookup ("what points to this node?") are both prefix scans. Write cost of maintaining both entries is negligible at AKG's write frequency. The alternative — a full scan for inbound edges — gets painful even at modest scale.

### 3.3 — Tag index
**Decision:** For every tag on a node, write `t:{tag}:{node_id}`. "All nodes tagged `auth`" is a prefix scan on `t:auth:`.

Tag rules:
- Lowercase only
- Snake_case for multi-word tags (e.g. `auth_flow`, `error_handling`)
- Maximum 32 tags per node
- Spaces rejected at write time, not normalized

**General principle established:** The SDK should be loud about bad input and never silently correct it. Fail fast, fail clearly. This carries forward through the entire spec.

### 3.4 — Temporal index
**Decision:** Temporal keys are self-describing and keyed on `updated_at`.

Canonical forms:
- `ts:{timestamp}:n:{type}:{id}`
- `ts:{timestamp}:e:{from}:{relation}:{to}`

This supports the natural agent access pattern of recency queries ("what did I learn last session?") while preserving full logical identity in the key itself. Timestamp is Unix microseconds, matching the field encoding decision from Step 2. One temporal index entry is written per logical record. Creation-time indexing is not part of v1.

### 3.5 — Title prefix index
**Decision:** Dropped entirely. Not a natural agent access pattern — agents are precise, they traverse edges or query by type/tag, not by remembered title fragments. Doesn't solve full-text search anyway. Can be added non-breakingly in a future version if genuinely needed.

### 3.6 — Complete key prefix table

| Prefix | Key structure | Purpose |
|--------|--------------|---------|
| `n:` | `n:{type}:{id}` | Node primary store, grouped by type |
| `e:` | `e:{from_node}:{relation}:{to_node}` | Outbound edge lookup |
| `ei:` | `ei:{to_node}:{relation}:{from_node}` | Inbound edge lookup |
| `t:` | `t:{tag}:{node_id}` | Tag inverted index |
| `ts:` | `ts:{timestamp}:n:{type}:{id}` or `ts:{timestamp}:e:{from}:{relation}:{to}` | Temporal index keyed on `updated_at` |

## Step 4 — Header and Section Table

### 4.1 — Magic bytes
**Decision:** First four bytes of every `.akg` file are `AKG\0` (three ASCII letters followed by a null byte). Any reader that does not see this exact sequence at position zero must reject the file immediately with a clear error. Not warn, not attempt recovery — reject.

### 4.2 — Version semantics
**Decision:** Header stores a major and minor version number with the following compatibility contract:

- **Minor version bump** (e.g. 1.0 → 1.1) — backwards compatible. New optional fields, new section types, new conventions. A v1.0 reader opening a v1.3 file must work correctly, ignoring what it doesn't understand.
- **Major version bump** (e.g. 1.x → 2.0) — breaking change. A reader must reject any file with a higher major version than it understands, with a clear error. Loud failure beats silent corruption.

### 4.3 — Checksum scope and failure policy
**Decision:** Two levels of checksumming:
- The header itself is checksummed (excluding the checksum field)
- Every section has its own checksum

A single byte in the header identifies the checksum algorithm in use:
- `0x01` = CRC32 (default — fast, lightweight, sufficient for accidental corruption detection)
- `0x02` = SHA-256 (future — cryptographic, tamper-evident, appropriate for enterprise/sensitive data)
- `0x03` = BLAKE3 (future — modern cryptographic, faster than SHA-256)

Stating the algorithm in the header is a best practice (algorithm agility), not a security concern. Security comes from the hash value, not algorithm secrecy.

**Failure policy:** Reject on any checksum failure — header or section. The SDK may provide an explicit `akg.recover()` rescue operation for salvaging readable sections from a corrupted file, but this is not normal operation.

### 4.4 — Section ordering
**Decision:** Sections may appear in any order in the file. The section table is the authoritative source of truth for section locations — readers must use it, not assume order. Exception: the header is always first, always at bytes 0-63, no exceptions.

Rationale: flexible ordering makes compaction simpler (sections can be written as ready) and costs nothing on a memory-mapped local file where random access is essentially free.

### 4.5 — Bloom filter
**Decision:** Bloom filter lives in its own dedicated section (not the header) so it can scale proportionally with file size. At 10 bits per key a 100k node file with ~500k total keys produces a ~625KB bloom filter — variable size, so it must be a section.

Fixed parameters (must be in the spec so every implementation produces identical filters):
- Hash function: MurmurHash3 x64 128
- Bits per key: 10
- Number of hash functions: 7
- Hash seed: 0
- Bit ordering within each byte: least-significant-bit first

The bloom filter is rebuilt from scratch on every compaction. Between compactions, new keys are added but deleted keys remain (slightly increasing false positive rate until next compaction — acceptable).

Purpose: fast negative lookups for the common agent pattern "do I already know about X?" A bloom filter miss means definitely no — skip the binary search entirely. A bloom filter hit means maybe yes — fall through to the real index.

### 4.6 — Header byte layout
**Decision:** The header is exactly 64 bytes, always at position 0. Field layout:

```
Bytes 0-3:   Magic (`AKG\0`)
Byte  4:     Major version (uint8)
Byte  5:     Minor version (uint8)
Byte  6:     Checksum algorithm (uint8)
Bytes 7-10:  Section count (uint32)
Bytes 11-54: Reserved (zero, for future use)
Bytes 55-58: Header checksum CRC32 (uint32)
Bytes 59-63: Reserved (zero)
```

The section table begins immediately at byte 64.

### 4.7 — Section table entry structure
**Decision:** The section table lives immediately after the header (byte 64), with one entry per section. Entry structure:

```
Section Table Entry {
  type:   uint8   // section type ID
  offset: uint64  // byte offset from start of file
  length: uint64  // byte length of the section
}
```

The number of entries is given by the section count field in the header. Readers navigate to any section using this table — order of entries is not assumed.

### 4.8 — Section type IDs
**Decision:** Each section is identified by a uint8 type byte in its section table entry. Defined values:

- `0x01` — Data (sorted key/value store)
- `0x02` — Bloom filter
- `0x03` — WAL

A reader encountering an unknown type ID must skip that section, not reject the file. This allows new section types to be added in a minor version bump.

### 4.9 — Section checksum structure
**Decision:** Each section's payload is followed immediately by its checksum bytes on disk. The checksum covers the payload bytes only. The `length` field in the section table includes the checksum bytes, so a reader that needs the payload alone subtracts the checksum size. The checksum algorithm is the one declared in the header algorithm byte.

## Step 5 — WAL Record Format and Replay Semantics

### 5.1 — WAL operation types
**Decision:** Five record types, no more:

- `PUT_NODE` (1) — create or update a node. No separate UPDATE — an update is a PUT with a new version.
- `DELETE_NODE` (2) — tombstone a node
- `PUT_EDGE` (3) — create or update an edge
- `DELETE_EDGE` (4) — tombstone an edge
- `COMMIT` (5) — marker that everything before this point has been fully applied and the file is consistent

Rationale: the append-and-tombstone write model means create and update are the same operation at the WAL level. Four mutation types plus a commit marker is the minimal complete set.

### 5.2 — WAL record structure
**Decision:** Every WAL record has the following fixed structure:

```
WAL Record {
  sequence:   uint64      // monotonically increasing, never reused
  operation:  uint8       // PUT_NODE=1, DELETE_NODE=2, PUT_EDGE=3, DELETE_EDGE=4, COMMIT=5
  length:     uint32      // byte length of the payload that follows
  payload:    bytes       // MessagePack-encoded node or edge (empty for COMMIT)
  checksum:   uint32      // CRC32 of everything above
}
```

Key properties:
- **Sequence number** is uint64 — effectively inexhaustible (18.4 quintillion values). Monotonically increasing, never reused, never resets. Establishes operation order and enables merge conflict detection.
- **Length prefix** enables detection of partial writes on crash — if remaining bytes don't match stated length, the record is incomplete and discarded. Clean recovery, no guessing.
- **Per-record checksum** means a corrupted record doesn't poison everything after it. Each record is independently verifiable.
- **Payload is MessagePack** — same encoding as Step 2. No new format introduced.

### 5.3 — WAL lifecycle and clearing
**Decision:** WAL accumulates between compactions and is discarded entirely when a fresh compacted file is written. No incremental clearing.

Safety valve: automatic flush triggered at **1,000 uncompacted entries or 10MB**, whichever comes first. Prevents unbounded WAL growth from a long-running agent that never triggers compaction naturally.

### 5.4 — fsync semantics
**Decision:** fsync on explicit `commit()`. The SDK buffers writes in memory until `commit()` is called — that is the durability boundary. Everything before `commit()` is fast but not guaranteed. Everything after is on disk.

- `commit()` is the natural end-of-agent-turn boundary
- `commit()` is automatically called on clean close
- If the process crashes before `commit()`, only that turn's work is lost — acceptable given the usage pattern

### 5.5 — WAL replay semantics (clarification)
**Decision:** The conventional WAL model applies. A `COMMIT` record closes a committed batch — everything before it is durable and intentional. On ordinary open, replay all records up to and including the last valid `COMMIT`. Trailing records after the last valid `COMMIT` are uncommitted and must be discarded, not applied. This is consistent with 5.4: a crash before `commit()` loses only that turn's work.

## Step 6 — Compaction

### 6.1 — Compaction trigger
**Decision:** Explicit only. Compaction runs when the caller explicitly calls `compact()`. The WAL flush threshold (1,000 entries or 10MB) is a safety valve against unbounded WAL growth, not a compaction trigger. The natural compaction boundary is end-of-session, which the caller controls.

### 6.2 — Compaction process
**Decision:** Compaction writes all live keys (tombstones excluded) into a new sorted file, rebuilds the bloom filter from scratch, then atomically renames the new file over the old one. The WAL is discarded at this point. A crash at any point during compaction is safe — the rename is atomic, so the file is either the old version or the new one, never a corrupt hybrid.

### 6.3 — Tombstone handling
**Decision:** Tombstones are dropped on compaction, full stop. The compacted file contains only live records. Note for the merge spec (Step 9): merge logic will need to account for the fact that a compacted file carries no record of deletions.

## Step 7 — Conformance Tests

### 7.1 — Conformance test corpus
**Decision:** A corpus of `.akg` binary files lives in the reference implementation repo (Go), with a note in the spec that it is the canonical cross-implementation standard. The corpus covers five categories:

- **Baseline cases:** empty graph, minimal node (required fields only), fully-populated node, single edge, small realistic graph (~10 nodes, ~15 edges, mix of types and tags)
- **Encoding edge cases:** node with all valid MessagePack value types in `meta`, node with maximum tags (32), node with a very large `body`, edge with `confidence: null` vs `confidence: 0.5`, edge with `strength` at exactly `0.0` and `1.0`
- **Format state cases:** file with uncompacted WAL requiring replay, compacted file (clean, no WAL), file with tombstoned nodes and edges
- **Rejection cases:** wrong magic bytes, unsupported major version, bad header checksum, bad section checksum, truncated WAL record

*Note: the corpus validates readers. Spec precision is the primary mechanism for keeping writers honest.*

## Step 8 — Reference Implementation

### 8.1 — Reference implementation scope
**Decision:** The reference implementation is built in two phases. Phase 1 covers the format layer only: read, write, compact, WAL replay during ordinary open, and explicit recovery tooling. This is sufficient to validate the spec and generate the conformance corpus. Phase 2, targeted at pre-wide-release, extends to the full SDK surface: `put_node`, `get_node`, `find_nodes`, `delete_node`, `put_edge`, `get_edges`, `delete_edge`, `merge`, and `stats`.

## Step 9 — Merge Semantics

### 9.1 — Merge philosophy
**Decision:** The spec defines conflict detection, not conflict resolution. The spec must define what constitutes a conflict (same node or edge identity, differing content) and what information must be preserved to allow resolution (both versions, flagged). Resolution strategy — last write wins, caller resolves, consolidation agent — is SDK policy, not format policy. The consolidation agent is an optional SDK-level concern, not a format requirement.

### 9.2 — Tombstone/merge gap
**Decision:** The spec acknowledges that compaction permanently erases tombstones (per 6.3), which creates an unresolvable ambiguity during merge: a record absent from a compacted file may have been deleted or may never have existed. Resolving this correctly likely requires a persistent deletion log that survives compaction. The design of this mechanism is deferred to Phase 2.
